# Setup manifest

Every repo generated by `archon-setup` gets `.github/archon-setup.json`. It is the answer to "what got installed where, and from what source?"

## Schema

```json
{
  "tool": "archon-setup",
  "toolVersion": "0.1.0",
  "createdAt": "2026-05-13T00:00:00.000Z",
  "owner": "ArchonVII",
  "repo": "example-repo",
  "visibility": "private",
  "sourceSnapshots": {
    "githubWorkflows": { "ref": "v1", "sha": "..." },
    "repoTemplate": { "ref": "main", "sha": "..." },
    "orgDefaults": { "ref": "main", "sha": "..." }
  },
  "selectedFeatures": [
    "foundation.readme",
    "foundation.agents"
  ],
  "onboardingDispositions": {
    "schemaVersion": 1,
    "items": [
      {
        "itemId": "foundation.readme:README.md",
        "feature": "foundation.readme",
        "path": "README.md",
        "status": "drifted",
        "choice": "keep-local",
        "decidedBy": "repo-owner",
        "decidedAt": "2026-07-18T00:00:00.000Z",
        "fingerprint": { "algorithm": "sha256", "value": "..." },
        "runId": "onboard-example",
        "baseSha": "...",
        "decisionSource": {
          "type": "github-issue",
          "owner": "ArchonVII",
          "repo": "example-repo",
          "number": 123,
          "url": "https://github.com/ArchonVII/example-repo/issues/123"
        }
      }
    ]
  },
  "createdFiles": [
    { "path": "AGENTS.md", "source": "snapshot:repo-template/AGENTS.md" }
  ],
  "remoteActions": [{ "type": "labels.apply", "result": "ok" }],
  "postChecks": [
    {
      "type": "branchProtection.tightenRequiredChecks",
      "deferUntil": "firstCheckRun",
      "status": "complete",
      "completedAt": "2026-05-13T01:00:00.000Z",
      "requiredCheck": "repo-required-gate / decision",
      "branch": "main",
      "result": "required"
    }
  ]
}
```

## Onboarding dispositions

Existing-repo repair stores validated owner decisions in the versioned
`onboardingDispositions` section. Legacy manifests without this section keep
their previous audit behavior.

- `keep-local` is accepted only while the repository file matches the recorded
  SHA-256 fingerprint. A later content change makes the item stale and
  actionable again.
- `declined` deselects the whole capability. Declined features are absent from
  the effective `selectedFeatures` array, while the disposition remains as
  provenance.
- `merge-manual`, `blocked`, and `defer` remain unresolved and keep onboarding
  incomplete. A defer must include `review.trigger` or `review.expiresAt`; an
  elapsed expiry or recorded `review.triggeredAt` is reported as actionable.
- `apply-central` records the decision that authorized the automated write. It
  does not excuse later missing or drifted content.

Disposition entries are replaced by `itemId` on later repair runs, so an owner
can explicitly supersede an earlier decision without accumulating conflicting
active records.

## Capability bits (used at plan-build time)

| Bit                          | Meaning                                                  |
| ---------------------------- | -------------------------------------------------------- |
| `gh.installed`               | `gh --version` succeeded                                 |
| `gh.authenticated`           | `gh auth status` reports authenticated to github.com     |
| `gh.activeAccountKnown`      | An account name was parsed from `gh auth status`         |
| `gh.repoCreateAllowed`       | Account can create repos under chosen owner              |
| `gh.repoAdminAllowed`        | Account has admin on the (eventual) repo                 |
| `gh.branchProtectionAllowed` | Implied by `repoAdmin`                                   |
| `gh.actionsSecretsAllowed`   | Account can set Actions secrets                          |
| `gh.orgAdminAllowed`         | Account is an org admin (only used by Copilot v0.4 path) |
| `gh.copilotAgentAllowed`     | Copilot cloud agent APIs accessible (v0.4)               |

## Why this exists

- "Audit existing repo" mode (v0.2) reads this manifest to know what to compare against.
- Future "Upgrade workflows" mode reads `sourceSnapshots.githubWorkflows.sha` to detect drift.
- Forensics: lets a future agent answer "did we ever wire X?" without grepping every file.
- Post-checks move from deferred to `status: "complete"` only after the follow-up action
  succeeds or is already present, such as `tighten-required-gate` making
  `repo-required-gate / decision` required.
