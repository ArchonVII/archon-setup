# 2026-07-01 - #317 Commit hooks executable on Windows onboards

- **Changed paths:** `src/server/tasks/initGitAndCommit.mjs`, `src/server/tasks/writeGithooks.mjs` (exports `HOOK_FILES`), `test/onboardHookExecBit.test.mjs` (new), `.changelog/unreleased/317-committed-hook-execbit.md`, this fragment.
- **What changed:** The end-to-end closure of the #294/#314 exec-bit arc. `writeGithooks.apply()` chmods `0o755` on disk, but nothing staged the git mode, so a Windows onboard (`core.filemode=false`) committed hooks as `100644` and a Unix clone silently skipped the guards. `initGitAndCommit.apply()` now runs `git update-index --add --chmod=+x -- <hook entrypoints>` between the bootstrap staging step and the bootstrap commit, filtered to hooks actually present (hook-less feature selections unaffected), cross-platform and idempotent on Unix.
- **Verification:** Red→green on this Windows host: with the fix stashed, the new test fails `.githooks/commit-msg must be committed executable (got 100644)`; with the fix, a real `runOnboard` into a temp dir commits all eight entrypoints at `100755` (`git ls-files -s`). Full suite `npm test` → 638 pass / 0 fail / 2 skipped on the lane worktree.
- **Propagation:** none — onboarding-engine behavior; already-onboarded repos with committed `100644` hooks need a one-time exec-bit restage in the Phase F backfill sweep (noted for W6).
